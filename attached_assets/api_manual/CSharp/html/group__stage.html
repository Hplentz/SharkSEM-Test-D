Detailed Description

Most microscopes are equipped with motorized stage. Stage can be fully controlled via remote control interface, there are only limitations related to the stage model. For example, some microscopes do not support motorized movement in Z or ROTATION axis. Only some microscopes are equipped with secondary tilt axis - TiltY

Get stage limits.

Device limits specify the stage travel range in arbitrary axis. Soft limits may be stricter that device limits, for example when there is a detector which prevents travel across the whole device range.

If an axis is not motorized, limits are zero.

Compatibility issues prevention: When tilt y is not motorized, 10 values are returned. When tilt y is motorized, 12 values are returned.

Parameters
Type0 - device limits, 1 - soft limits

Get stage motorization.

Most microscope models use 5-axis motorization, however, there are instruments with special manipulators. It can also happen that arbitrary axis is switched off.

Compatibility issues prevention: When tilt y is not motorized, 5 values are returned. When tilt y is motorized, 6 values are returned.

Return values
MotorizedX0 - axis is motorized, 1 - axis is not motorized
MotorizedY0 - axis is motorized, 1 - axis is not motorized
MotorizedZ0 - axis is motorized, 1 - axis is not motorized
MotorizedRot0 - axis is motorized, 1 - axis is not motorized
MotorizedTiltX0 - axis is motorized, 1 - axis is not motorized
MotorizedTiltY(optional) 0 - axis is motorized, 1 - axis is not motorized
Timing
Executed immediately.

◆ StgMove() [1/6]

static void SharkSEM.Commands.StgMove ( this Connection  Conn,
double  X 
)
inlinestatic

Start stage movement, specify velocity.

Infinite movement is started. Application must stop it later using StgStop() call.

If less than 5 arguments are passed, the other velocities are zero.

This function is supported by all types of stages. It is recommended to use watchdog timer together with StgMove(). Refer to StgWatchdog().

Warning
Stages without position indicators will run infinitely. If the stage is forced to move for a long time after it hits one of the mechanical limits, it can be damaged.
Parameters
Xx velocity [mm/s]
Yy velocity [mm/s], optional
Zz velocity [mm/s], optional
Rotrotation velocity [degrees/s], optional
TiltXtilt x velocity [degrees/s], optional
TiltYtilt y velocity [degrees/s], optional
Timing
Executed immediately. But movement takes some time. Wait B flag is set during the movement.
See also
StgWatchdog()
StgStop()
StgIsBusy()
Call Context
Stage must be idle. If the stage is busy, the command is ignored.
Version
1.0.10 and later

◆ StgWatchdog()

static void SharkSEM.Commands.StgWatchdog ( this Connection  Conn,
double  Timeout 
)
inlinestatic

Stage safety watchdog timer.

Stage watchdog timer is a safety feature. When stage movement is started from remote application, it may be hazardous if the remote application crashes or if the connection is lost, because the movement won't stop.

Before starting the movement, application may schedule watchdog timer, which stops the movement when it times out. The application must periodically restart the timer, indicating that it is still alive.

The timer is restarted by calling StgIsBusy. When StgIsBusy reports "in motion" status, timer is internally restarted. When it reports "stopped" status, the timer is killed.

Once StgWatchdog() is initiated, application is responsible for calling StgIsBusy periodically, in a shorter time than specified by timeout. The timer is killed either if StgIsBusy reports that the movement has been finished or if StgStop() is called explicitly.

Example:

StgWatchdog(2.0); // schedule watchdog timer to 2 s
StgMoveTo(10, 20); // movement command
while (StgIsBusy()) { // restart/kill the timer
if (stop_requested) { // if stop requested by user
StgStop();
break;
}
Sleep(1000); // wait a while
}
Parameters
Timeouttimeout [s]
Timing
Executed immediately.
Call Context
Just before movement is started.
Version
2.0.22 and later
