Rotating Chord Algorithm

This chapter describes interface of RCA (Rotating Chord Algorithm) extension. This extension has been developed in cooperation with U.S. National Institute of Standards and Technology.

The Rotating Chord Algorithm (RCA) functions are invoked in the following order:

  1. RCAInit()
  2. RCANextParticle()
  3. RCANextParticle()
  4. …
  5. RCAFinish()

Besides the three main functions, there are a few additional functions:

  • RCAGetDACRange()
  • RCASetCbMask()
  • RCASetOption()
  • RCASkipParticle()

The SharkSEM server informs the client about the progress and results of the analysis by issuing callbacks. Pixel coordinates, image and synchronization stamps are sent in these callbacks.

This document describes the interface functions (SharkSEM API), the details about RCA implementation are not discussed.

Particle search

The particle search process consists of several steps:

  1. Scan single horizontal line, identify candidate pixels whose brightness is greater or equal to search_thr_low and less than or equal to search_thr_high.
  2. Process the pixels from left to right, if there is a candidate which does not belong to an existing object (particle), start RCA characterization.
  3. The RCA characterization uses different brightness range – search_thr_low, search_thr_high, and also different dwell time – meas_dwell. It consists of centroid search, shape characterization and finally X-Ray analysis.

During search a Previous Particle Map (PPM) is used. PPM is rectangular, its width (height) is power of two. Pixel mask of all detected objects is stored in PPM, its main purpose is to prevent duplicate particle detections.

Modes

RCA supports severeal particle scanning strategies - modes. They are used by particle functions as RCANextParticle().

Mode 1
In the mode 1, beam is placed in the center of the particle for the specified time. Following additional parameters are effective:
  • EdxDwell
Mode 2
In the mode 2, beam scans rectangular pixel mesh over the particle. The scanning is done in regular XY mesh.
The scanning uses zig-zag mode (first pixel row forth, second back, third, forth, etc.) Number of rows is rounded to the closest even number. Following additional parameters specify the mesh:
  • EdxDwell
  • EdxMeshWidth
  • EdxMeshEnlargeMin
  • EdxMeshEnlargePerc
Mode 3
Mode 3 is similar to mode 2. The only difference is the mesh spacing calculation. In mode 2, there is fixed bitmap size (width or height, whichever is greater). In mode 3, the pixel spacing is specified instead.
Following additional parameters are effective:
  • EdxDwell
  • EdxMeshSpacing
  • EdxMeshEnlargeMin
  • EdxMeshEnlargePerc

Both mode 2 and mode 3 use zig-zag scanning of rows, odd frames are scanned from top to bottom, even frames bottom up. The electron beam moves continuously, there are no flybacks. The pixel mesh dimensions are based on bounding rectangle of the particle, margin specified by the additional parameters is added. In the mode 2, EdxMeshWidth is number of pixels per longer dimension (width or height).

In both mode 2 and mode 3, the dimensions of the pixel mesh can be determined from status event 70, parameters p2 and p3.

Mode 4
Mode 4 has been implemented in SharkSEM version 2.0.23. It scans over the particle similar “stars” like in the particle measurement phase. Each star, the particle center is evaluated and re-centering takes place.
Following parameters define the algorithm behavior:
  • EdxDwell
  • EdxMeshSpacing
  • EdxNumChords
  • EdxThrLow
  • EdxhrHigh

EdxDwell is the pixel dwell time in bright regions, it is typically set to tens or hundreds of µs. In dark regions (outside particle), dwell time is set to meas_dwell. Total time of analysis is given by edx_param.

The total time can be slightly exceeded, because integer number of complete stars must be scanned.

Callbacks

There are three callback functions:

  • RCACbSync() – processing status information, particle size & position information
  • RCACbCoords() – beam position list
  • RCACbImage() – pixel brightness list

The callback messages are sent through data connection. Since the volume of data is relatively high, client must process the callbacks in an efficient way.

If the client does not process the data quickly enough, they are buffered on the server side. This means no data gets lost, and inability to process the data does not slow down the RCA analysis.

RCACbCoords

Callback specifying pixel coordinates.

Each scanned pixel is assigned unique identification. The identification consists of seq_id and pixel index. When the complete image frame starts, the seq_id is zero and pixel index as well. As scanning goes on, pixel index is sequentially increased. At some point (in synchronization point), seq_id is incremented and pixel index is reset to zero.

This mapping is the same for both SEM data and EDX data, because SEM sends synchronization signals to Pulsetor EDX DPP (Digital Pulse Processor). In the Pulsetor documentation, seq_id is called frame_id instead.

To make transfer of coordinates more efficient, only pixel index of the first pixel in the array (pxl_first) and number of coordinates in the array (pxl_count) are transferred.

In each 32-bit word, the high order 16-bits specify the DAC-X position, low order 16-bits specify the DAC-Y position.

Typical client implementation is that it stores the mapping in a table. It is guaranteed that coordinates arrive prior to the corresponding image, and when image packet is received, coordinates of each pixel are already known.

However, this principle cannot be applied to X-Ray data received from DPP. Because DPP and SEM run asynchronously, there must be X-Ray data FIFO buffering X-Ray energies. The energies can be processed only if client already knows their seq_id.

RCACbSync

Synchronization callback.

All important events like line start, RCA procedure start, particle dimensions, etc. are recorded and sent to the client. This callback is synchronous with image data, i.e. client should typically observe the reported state and process the data according to the state.

event Description Parameters
0 Image frame started (-, -, -, -)
10 Scan line started (l_ind, -, -, -)
20 Scan line completed (l_ind, -, -, -)
30 Centroid search started (p_ind, -, -, -)
31 Centroid chord completed (x1, y1, x2, y2)
40 Centroid search completed (p_ind, cx, cy, -)
50 Measurement started (p_ind, -, -, -)
51 Measurement chord completed (x1, y1, x2, y2)
60 Measurement completed (p_ind, m_stat, -, -)
70 EDS mapping started (p_ind, width, height, -)
71 EDS re-centering done (p_ind, cx, cy, -)
80 EDS mapping completed (p_ind, -, -, -)
90 Image frame completed (-, -, -, -)

Parameters:

  • l_ind: PPM row index
  • x1, y1: start coordinates of a chord [DAC steps]
  • x2, y2: end coordinates of a chord [DAC steps]
  • cx, cy: center of the particle [DAC steps]
  • p_ind: particle index
  • m_stat: particle measurement status, 0 – particle is valid, 1 – particle is a duplicate of another particle
  • width: width of the EDX measurement mesh (pixels)
  • height: height of the EDX measurement mesh (pixels)
